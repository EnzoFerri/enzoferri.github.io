Form

Poche le personalizzazioni fatte per i form.
Le uniche regole specifiche inserite nel file lomb_ver.css riguardano le classi radio e checkbox, per disporre le relative label coerentemente con le linee guida della Regione Lombardia. 

.radio label,
.checkbox label {
  display: block;
}
.radio input[type="radio"],
.radio-inline input[type="radio"],
.checkbox input[type="checkbox"],
.checkbox-inline input[type="checkbox"] {
  float: none;
  margin-left: 0;
}
E' stata personalizzata la resa grafica del tag <hr> per renderlo visibile rispetto allo sfondo. 
hr {
  border-style: inset;
}

La strutturazione della videata va quindi eseguita utilizzando il Grid system di bootstrap, che permette di ottenere un layout responsivo.

Campi data

Al momento il type="date" per i campi di input non è correttamente supportato da tutti i browser.
Chrome, tra i più diffusi, dà la possibilità sia di digitare direttamente la data in una maschera del tipo gg/mm/aaaa o di cliccare sul campo ed ottenere un calendario con cui interagire e scegliere una data.
Al momento può essere più opportuno non specificare tale tipo dati e utilizzare javascript per il controllo della data e l'emissione eventuale di un calendario.

Accessibilità

In questi template per i form non sono stati inseriti attributi o tag specifici per l'accessibilità.
Indispensabile è legare la giusta label ad ogni campo di input, tramite l'attributo for per il tag <label> ed il corrispondente attributo id dal tag <input>. 
<label class="control-label" for="cognome">Cognome</label>
<input id="cognome" class="form-control" type="text"/>

Leggendo la pagina con alcuni screen-reader non si sono riscontrate particolari difficoltà ad eccezione dei campi con attributo type="date", di cui sopra.
Per questi l'importante è che sia possibile digitare la data direttamente da tastiera, ed ottenere un eventuale messaggio di errore se errata.
Non deve avvenire automaticamente la comparsa di un calendario.

L'accessibilità effettiva della pagina dipende molto anche dalla disposizione dei campi a video.
E' quindi necessario, dopo aver passato l'analisi con il Web Accessibility Checker, eseguire un test con uno screen reader sulla pagina creata.
